The Numerous Pains of Programming: Death by a Thousand Cuts

October 25, 2024

I recently started building a small Python application, comprising less than 1000 lines of code. However, it’s been a real challenge to get it to the point where it can be deployed and run in a production environment. I’ve done this before, but it’s always difficult, and each time I’m reminded of how much effort it is, in part because I always have to start anew.

As a part-time developer…which of these extra things do I actually have to do? And to what extent? The code, sure, but all the other things?

The Pain of Programming

For most applications, writing the code is relatively straightforward, though it can still be challenging. However, there are tons of other things around the code, all that extra work you have to do to make it a quote unquote “real application”.

Below I’m not even including things like picking a language or framework.

NOTE: This is not a perfect, comprehensive list–it’s a brainstorm of things I can think of that I’ve had to do, or might have to do to make an application “real” and somewhat professional.

  1. Writing and maintaining comprehensive test suites, unit tests, integration tests, etc.
  2. Setting up testing infrastructure and testing frameworks
  3. Designing and implementing CI/CD pipelines
  4. Creating and maintaining distributable packages
  5. Managing production environment deployment and monitoring
  6. Implementing release management
  7. Writing and maintaining documentation
  8. Creating and updating system architecture diagrams
  9. Managing version control workflows and git complexity
  10. Supporting and responding to user needs and feedback
  11. Implementing basic security best practices
  12. Conducting stress testing and performance optimization
  13. Containerizing the application and creating Kubernetes manifests
  14. Setting up vulnerability scanning
  15. Managing security issues in dependencies
  16. Managing databases, data models, and migrations
  17. Configuring IDE settings and extensions
  18. Setting up development environments (e.g., virtual environments)
  19. Managing dependency trees and version conflicts
  20. Configuring and maintaining linters and formatters
  21. Setting up logging and monitoring infrastructure
  22. Dealing with debugging tools…more than print statements?
  23. Testing with SSL certificates
  24. Deploying on different operating systems and user environments
  25. Dealing with authentication, perhaps RBAC too
  26. Do I need different environments for development, testing, staging, and production?
  27. Keeping track of issues, tickets, and other project management tasks
  28. How to get secrets and configuration into the application safely and securely
  29. Dealing with updating the software, from configuration to removing features and functionality
  30. Understanding resource requirements, memory, cpu, etc. and what to set limits to in production
  31. Threat modelling–how would an attacker get in, what would be the impact, etc.
  32. What license should I use for the software? What does it mean if I choose the wrong one?
  33. What metrics should it output, and how?

I think I could keep going, and I’m sure there are many more.

The above is all a tremendous amount of work, work that extends well beyond the few hundred or thousand lines of code you write for the core of a simple application. For a small application, it doesn’t seem worth it. There doesn’t seem to be much in the way of easy-to-use automation or other tools to help deal with it all–it’s just a big old mess of tasks that add up to a lot of work.

What is a Real Application?

I keep using the phrase “real application”, but what does that mean? What makes an application real? I suppose what I mean is an enterprise, production-grade application, of some kind. But that doesn’t mean much–I’m not sure anyone knows what makes an application production-grade. It’s kind of a vague term, and I think we could probably do better. The reality is that programming is more of an art, a dedicated craft, with hundreds of things that need to be done and hundreds many more tools, technologies, and techniques that can be used to do them. Some of this is optional, some of it is required, and some of it is just good practice. Which is which…it’s not always clear.

Tedious Work

I find all this extra work absolutely exhausting and, frankly, quite tedious. In every single one of these steps it seems to me like there just has to be a better way. And for some, I’m sure there is, I simply may not know about it yet.

Ultimately, I’d love to have a magic wand to wave and get all this done, and thus just focus on the code that provides the value, but I don’t think that’s possible. Generative AI isn’t going to help with this all that much. Imagine what we could do if we just got rid of all this tedium?

An Application Can Never Be Done, but It Has to Be Run

I think the reality is that an application can never be complete. There is always more work. It can never be 100% finished–there is always something missing, which is an incredible place to be when you consider that one of humanity’s main outputs at this point in time, one of our main economic drivers, is the writing and running of software. What a bizarre situation to be in!

My next post will be regarding some solutions to the pain of programming, some kind of protective armor that can stop the thousand cuts.
